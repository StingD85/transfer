## Root cause of the parameter visibility bug

`MR_PARAMETERS.txt` carried 73 placeholder strings in the GUID column instead of valid 8-4-4-4-12 hex UUIDs. Examples:

```
PARAM	v4-0001-0000-0000-000000000006	ASS_FAB_STATUS_TXT	TEXT	...
PARAM	v6-0001-0000-0000-000000000017	COMM_STATE_TXT	TEXT	...
```

At runtime `ParamRegistry.LoadFromFile()` reads the file (line 1391-1402 of `ParamRegistry.cs`) and uses `Guid.TryParse()` on each row. The `v4-` and `v6-` prefixed strings fail TryParse silently (line 1398), so those parameters never enter `_guidByName`. Revit then has no GUID for them, so Edit Label dialogs and shared-parameter pickers cannot bind them.

The 73 affected parameters span 142 of 142 tag families (every family touches at least one).

## Repair strategy

Each malformed GUID is replaced with a deterministic UUIDv5 keyed on the parameter name, using the standard URL namespace `6ba7b810-9dad-11d1-80b4-00c04fd430c8`.

Properties:

- Same input name always yields the same UUID. Re-running the repair is idempotent.
- New GUIDs are guaranteed valid hex.
- The mapping is reproducible without my involvement:

```python
import uuid
NS = uuid.UUID("6ba7b810-9dad-11d1-80b4-00c04fd430c8")
uuid.uuid5(NS, "ASS_FAB_STATUS_TXT")
# -> 376dd8ef-e6d7-552c-a5ba-1c3fe9fa6457
```

C# equivalent for any verification you want to do server-side:

```csharp
using System.Security.Cryptography;
using System.Text;

static Guid Uuid5(Guid ns, string name) {
    var nb = ns.ToByteArray();
    Array.Reverse(nb, 0, 4); Array.Reverse(nb, 4, 2); Array.Reverse(nb, 6, 2);
    var input = nb.Concat(Encoding.UTF8.GetBytes(name)).ToArray();
    using var sha = SHA1.Create();
    var h = sha.ComputeHash(input);
    var b = new byte[16]; Array.Copy(h, 0, b, 0, 16);
    b[6] = (byte)((b[6] & 0x0F) | 0x50);
    b[8] = (byte)((b[8] & 0x3F) | 0x80);
    Array.Reverse(b, 0, 4); Array.Reverse(b, 4, 2); Array.Reverse(b, 6, 2);
    return new Guid(b);
}

var ns = new Guid("6ba7b810-9dad-11d1-80b4-00c04fd430c8");
var g = Uuid5(ns, "ASS_FAB_STATUS_TXT");
// 376dd8ef-e6d7-552c-a5ba-1c3fe9fa6457
```

## Migration impact for live Revit projects

Revit identifies shared parameters by GUID. Changing a GUID under the same name produces a new parameter from Revit's perspective.

Practical impact for the 73 repaired parameters: **negligible**. The malformed GUIDs were never accepted by Revit, so any binding, schedule, filter, or tag that appeared to reference them was silently broken. There is no live project data tied to the old GUIDs because the bug prevented the binding from succeeding in the first place.
